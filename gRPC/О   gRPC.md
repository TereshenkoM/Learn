

gRPC (Remote Procedure Call) - фреймворк для удалённых вызовов процедур. Позволяет клиенту и серверу общаться напрямую, используя HTTP2 и Protocol Buffers (protobuf) в качестве языка описания интерфейсов (IDL).

###  Основные компоненты  gRPC

1. Protocol Buffers (protobuf). Язык описания данных + инструмент сериализации, используется для определения сервисов и их методов, а также для обмена данными между клиентом и сервером. Структура описывается в файле `.proto` и далее компилируется в исходный код выбранного языка.

2.  HTTP2. Протокол транспортного уровня, который обеспечивает мультиплексирование запросов, сжатие заголовков и т.д. (для улучшения производительности). Позволяет отправлять сразу несколько заголовков через одно соединение.

3. Stub генерация. gRPC автоматически генерирует клиентские и серверные стабы на основе протобаф файлов, что упрощает процесс интеграции и уменьшает количество шаблонного кода. Клиенты используют сгенерированные стабы для вызова методов на сервере (тк они бы вызывали локальные функции).


### Для  чего нужен?

В основном используется для построения микросервисной архитектуры и распределённых систем.

1. Микросервисы. Обеспечивает очень быстрое, эффективное, надёжное общение между сервисами 

2. Взаимодействие между разными языками. gRPC позволяет писать системы, где компоненты написаны на разных языках.

3. Реализация API. gRPC идеально подходит для создания высоконагруженных API. Где критичны низкие  задержки и высокая пропускная способность.

4. Мобильные и IoT приложения. Идеально подходит благодаря своей эффективности и низкому потреблению ресурсов.

### Преимущества и недостатки gRPC

+
1. Высокая производительность.
2. Ясно определённые интерфейсы
3. Поддержка различных языков
4. Би-ди стриминг
5. Автоматическая генерация кода

-
1. Ограниченная поддержка браузера
2. Зависимость от protobuff
3. Инфраструктурные требования
